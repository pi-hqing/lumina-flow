## Step 1 · 明确边界

### 1.1 Widget 的定位

Widget 是主应用的**轻量操作层**，以常驻悬浮窗形式附着于用户当前工作环境，核心价值是"不打断、不跳转、快完成"。它不是主应用的缩小版，而是主应用能力的高频子集入口。

| 维度 | Widget | 主应用 |
|------|--------|--------|
| 启动方式 | 快捷键 / 悬浮按钮唤起 | 独立窗口打开 |
| 推荐会话深度 | 1 ~ 3 轮；超过 3 轮可继续，但系统应评估是否触发 Escalation | 不限 |
| 任务复杂度 | 轻任务（< 2 分钟可完成） | 全任务 |
| 注意力占用 | 边缘注意力 | 主注意力 |
| 持久化需求 | 会话结果可选同步至主应用 | 完整持久化 |

### 1.2 Widget 的四项核心职能

**① 快速问答**  
用户在当前工作上下文中提出单次问题，期望在不切换窗口的前提下获得答案。典型场景：查术语、问公式、翻译短文本。

**② 轻任务执行**  
可在 Widget 内完成闭环的短流程操作。典型场景：创建一条待办、发送一条简短消息、触发一个预设自动化动作。

**③ 上下文感知提醒**  
Widget 基于当前应用、时间、日历或用户行为主动推送相关信息，无需用户主动唤起。典型场景：会议即将开始提醒、文档未保存警告、任务截止预警。

**④ Escalation 入口**  
当任务超出 Widget 能力边界时，触发 Escalation，将当前上下文无损传递至主应用，而非强行在 Widget 内完成。

> **术语约定**
> - **Escalation（升级至主应用）**：系统判断或用户选择，因任务超出 Widget 边界而打开主应用，并携带当前会话上下文。规范层统一使用此术语。
> - **Navigation（导航）**：用户主动跳转至主应用的特定页面（如点击历史记录），不携带 Escalation 语义。
> - **界面文案**：两种路径的按钮均可使用「打开主应用」，但规范内部须区分 Escalation 与 Navigation。

### 1.2.1 非目标（Non-Goals）

Widget **不追求**：

- 替代主应用
- 承载完整工作流
- 展示复杂信息架构
- 成为长期阅读界面
- 成为多任务工作台

> **设计原则：Widget 的目标是缩短任务开始时间，而不是缩短所有任务完成时间。**

当出现以下需求时，可直接以此边界拒绝：

```
把任务中心放进去
把知识库放进去
把审批页放进去
```

### 1.3 能力边界

**明确包含（Widget 范围内）**

- 悬浮入口与唤起交互
- 迷你会话（推荐 1 ~ 3 轮）
- 轻任务创建与状态确认
- 通知接收与快捷响应
- 跨窗口上下文感知
- 多显示器 / 多虚拟桌面行为

**明确不包含（属于主应用）**

- 完整对话历史管理
- 知识库浏览与编辑
- 复杂工作流配置
- 后台治理与权限管理界面
- 多步骤表单与审批流

**不可逆操作禁止原则**

涉及以下高风险行为时，Widget **不提供最终执行入口**，一律强制触发 Escalation：

- 批量修改 / 批量删除
- 权限变更
- 资金操作
- 正式发布

| 操作类型 | Widget 内允许 | Widget 内不允许 |
|----------|--------------|----------------|
| 删除确认 | 预览受影响范围、查看详情 | 最终提交执行 |
| 权限变更 | 查看当前权限状态 | 确认提交变更 |
| 正式发布 | 预览发布内容 | 点击发布按钮 |

**灰色区域（需根据场景判断）**

| 能力 | 默认策略 |
|------|----------|
| 长会话（> 3 轮） | 继续允许，系统评估是否建议 Escalation |
| 长文本生成 | Widget 预览摘要，主应用完整编辑 |
| 文件分析 | Widget 发起，主应用查看完整结果 |
| 多步骤 Agent 任务 | Widget 查看进度与状态，主应用管理任务 |
| 历史记录 | 最近记录可在 Widget 内查看，完整历史 Navigation 至主应用 |

### 1.4 与主应用的关系模型

```
主应用（完整能力层）
    ↑  Escalation（携带会话 ID、草稿、操作意图）
    ↑  Navigation（用户主动跳转，不携带 Escalation 语义）
    |
Widget（轻量操作层）← 快捷键 / 悬浮按钮唤起
    ↑
当前工作环境（用户正在使用的任意应用）
```

Widget 与主应用共享同一套 Agent 后端与数据层，但 Widget 侧仅订阅所需的 WebSocket 事件子集，不加载主应用完整状态树。Escalation 触发时，Widget 将当前会话 ID、输入草稿与操作意图一并传递，主应用无缝接续，用户无感知断层。

### 1.5 Escalation 判定规则（Escalation Heuristics）

满足以下**任意一条**，系统应触发 Escalation 建议：

| 条件 | 阈值 | 限制类型 |
|------|------|----------|
| 预估任务时长 | > 2 分钟 | 软限制 |
| 用户输入长度 | > 200 字符 | 软限制 |
| 预估输出长度 | > 500 字 | 软限制 |
| 交互轮次 | > 3 轮 | 软限制 |
| 并列信息面板数 | > 3 个 | 软限制 |
| 需要精确编辑 | 是 | 软限制 |
| 依赖完整历史上下文 | 是 | 软限制 |
| 涉及不可逆操作 | 是 | **硬限制** |

**软限制**：超出阈值时给出建议提示，由用户决定是否 Escalation。  
**硬限制**：强制触发 Escalation，不提供继续选项。

**Agent 响应示例（软限制）：**
```
用户：帮我写季度总结

Agent：这个任务预计内容较长，建议在主应用中完成，
      可以保留完整编辑和格式控制。

      [继续在这里开始]    [打开主应用]  ← Escalation
```

**Agent 响应示例（硬限制）：**
```
用户：删除这 10 条记录

Agent：批量删除属于不可逆操作，
      请在主应用中确认后执行。

      [打开主应用]  ← Escalation（无「继续」选项）
```

### 1.6 Widget 存在感规则

> **核心原则：Widget 的存在感应与任务相关性成正比，与用户当前专注程度成反比。**

#### 何时出现

**主动出现（用户触发）**
- 快捷键呼出
- 点击悬浮入口按钮
- 从通知点击展开

**被动出现（系统触发）**  
被动出现须**同时满足**以下三个前提：

```
1. 有明确的用户收益（不是系统状态播报）
2. 当前用户不处于全屏 / 演示 / 专注模式
3. 该类通知未被用户或企业策略静默
```

#### 何时消失

**立即消失**
- 用户点击关闭 / 按 Esc
- 用户点击 Widget 以外区域（默认开启，可配置关闭）
- Escalation 完成，主应用已打开

**延迟消失**
- 任务完成后停留 3 秒，显示完成状态后收起
- Agent 执行期间不消失，执行完成后进入延迟消失流程

**不消失**
- 用户正在输入时
- Agent 正在响应时
- 有未读通知未处理时（折叠态保留角标）

#### 何时打断用户

允许打断须**同时满足**：

```
事件优先级 = High
  AND
用户当前状态 ≠ 专注模式 / 全屏 / 演示
  AND
该类打断未被用户或企业策略静默
```

**High 优先级事件示例：**
- 会议即将开始（< 5 分钟）
- 有人 @ 用户且标记为紧急
- Agent 任务执行失败需要人工介入
- 权限即将过期影响当前操作

**永远不打断的场景：**
- 用户处于全屏模式
- 检测到正在进行视频通话 / 屏幕共享
- 用户已设置免打扰
- 企业策略禁止该类推送

#### 何时保持沉默

**静默但保留状态（折叠态角标）**
- 低优先级通知到达
- Agent 后台任务进度更新
- 非紧急的系统状态变化

**完全静默（不显示任何提示）**
- 用户处于专注模式期间的所有非紧急事件
- 重复性信息（同类通知 5 分钟内已推送过）
- 用户刚关闭 Widget 后的 30 秒冷却期内

#### 存在感优先级矩阵

| 用户状态 | High 优先级事件 | Medium 事件 | Low 事件 |
|----------|---------------|-------------|----------|
| 普通使用 | 打断展开 | 折叠角标 | 完全静默 |
| 专注模式 | 折叠角标 | 完全静默 | 完全静默 |
| 全屏 / 演示 | 完全静默 | 完全静默 | 完全静默 |
| 免打扰 | 完全静默 | 完全静默 | 完全静默 |

**专注模式判定（D-01 已确认）：**  
系统 API（macOS Focus Mode / Windows Focus Assist）OR Widget 手动开关，任意一个为真则进入专注模式，两者均关闭时退出专注模式。

## Step 2 · 用户与场景

### 2.0 设计前提

Widget 的用户不是某一类人，而是**处于专注状态的人**。

任何人在以下时刻都是 Widget 的目标用户：

```
正在做一件事
↓
遇到一个需要解决的小问题
↓
不想离开当前工作
↓
快速解决
↓
回到原来的事
```

### 2.1 用户状态模型

| 状态 | 描述 | 对 Widget 的容忍度 |
|------|------|-------------------|
| **深度专注** | 正在撰写、编码、设计等需要持续注意力的工作 | 极低，任何打断都有代价 |
| **轻度专注** | 浏览信息、整理文件、回复消息等可被短暂打断的工作 | 中等，可接受 < 30 秒的打断 |
| **任务间隙** | 刚完成一件事，还未开始下一件事 | 高，此时主动唤起 Widget 最自然 |
| **等待状态** | 等待他人回复、等待任务执行结果 | 高，此时被动提醒接受度最高 |

> **设计原则：Widget 主动触达的最佳时机是任务间隙与等待状态，深度专注期间应完全沉默。**

### 2.2 核心用户画像

#### 画像 A · 快速获取型

**核心特征：** 遇到问题，需要一个答案，拿到答案立刻回去继续工作。  
**典型工作状态：** 轻度专注 / 任务间隙  
**高频场景：** 写文档时确认术语、阅读英文时翻译段落、开会前查背景信息  
**挫败点：** 回答太长需要滚动、被引导进入多轮对话、拿到答案后 Widget 不自动收起

#### 画像 B · 轻任务执行型

**核心特征：** 有一个明确的小任务，希望不切换上下文就能执行。  
**典型工作状态：** 轻度专注 / 任务间隙  
**高频场景：** 记录待办、发简短消息、触发例行自动化  
**挫败点：** 操作步骤超过 3 步、完成后没有明确反馈、需要填写过多字段

#### 画像 C · 上下文响应型

**核心特征：** 不主动唤起 Widget，而是被提醒触达，需要快速判断并响应。  
**典型工作状态：** 等待状态 / 任务间隙  
**高频场景：** 会议提醒、Agent 任务完成通知、紧急消息确认  
**挫败点：** 提醒内容模糊、提醒频率过高、点击后进入复杂流程

### 2.3 核心用例

| # | 用例 | 画像 | 触发方式 | 预期完成时间 |
|---|------|------|----------|-------------|
| UC-01 | 快速问答（单轮） | A | 主动唤起 | < 15 秒 |
| UC-02 | 短文本翻译 | A | 主动唤起 | < 10 秒 |
| UC-03 | 上下文术语解释 | A | 主动唤起 | < 15 秒 |
| UC-04 | 创建待办事项 | B | 主动唤起 | < 20 秒 |
| UC-05 | 发送简短消息 | B | 主动唤起 | < 30 秒 |
| UC-06 | 触发预设自动化 | B | 主动唤起 | < 10 秒 |
| UC-07 | 会议提醒响应 | C | 系统触发 | < 10 秒 |
| UC-08 | Agent 任务完成通知 | C | 系统触发 | < 15 秒 |
| UC-09 | 紧急消息快捷回复 | C | 系统触发 | < 20 秒 |
| UC-10 | 长任务发起 + Escalation | A/B | 主动唤起 | < 15 秒（发起）|

### 2.4 用户旅程

#### 旅程 1 · 主动唤起 · 快速问答（UC-01）

```
用户处于专注状态
        ↓
遇到需要解决的问题
        ↓
快捷键唤起 Widget          ← 打断代价：< 1 秒
        ↓
输入问题
        ↓
Agent 响应                 ← 目标响应时间：< 3 秒
        ↓
获取答案
        ↓
Widget 自动收起（3 秒后）
        ↓
回到专注状态               ← 总打断时长目标：< 20 秒
```

**关键体验指标：**
- 唤起到可输入：< 300ms
- Agent 首字响应：< 1s
- 答案获取到 Widget 收起：用户无需额外操作

#### 旅程 2 · 系统触发 · 上下文提醒响应（UC-07）

```
用户处于轻度专注 / 等待状态
        ↓
Widget 角标出现（不打断）
        ↓
用户注意到角标，点击展开    ← 用户主动决定响应时机
        ↓
提醒内容直接呈现行动选项
        ↓
1 步操作完成响应
        ↓
Widget 收起
        ↓
回到原工作状态             ← 总打断时长目标：< 15 秒
```

#### 旅程 3 · 主动唤起 · 任务超出边界触发 Escalation（UC-10）

```
用户处于任务间隙
        ↓
快捷键唤起 Widget
        ↓
输入任务请求
        ↓
Agent 判断触发 Escalation 条件
        ↓
Widget 展示 Escalation 提示
（软限制：提供继续 / 打开主应用两个选项）
        ↓
用户选择「打开主应用」
        ↓
主应用打开，携带当前会话上下文
        ↓
Widget 收起
        ↓
用户在主应用中继续任务      ← 上下文无断层
```

### 2.5 场景优先级

**P0 · 核心场景（MVP 必须覆盖）**  
UC-01 快速问答、UC-04 创建待办、UC-07 会议提醒响应、UC-10 长任务 Escalation、UC-12 不可逆操作拦截

**P1 · 重要场景（MVP 后首批迭代）**  
UC-02 短文本翻译、UC-05 发送简短消息、UC-08 Agent 任务完成通知、UC-11 多轮对话 + Escalation 建议

**P2 · 扩展场景（后续版本）**  
UC-03 上下文术语解释、UC-06 触发预设自动化、UC-09 紧急消息快捷回复

## Step 3 · 信息架构（IA）

### 3.1 三层状态模型

```
┌─────────────────────────────────────┐
│          Layer 3 · 主应用           │  完整能力
│     Escalation / Navigation 触达    │
└─────────────────┬───────────────────┘
                  ↑ Escalation / Navigation
┌─────────────────┴───────────────────┐
│          Layer 2 · 展开态           │  边缘注意力
│     迷你会话 / 轻任务 / 通知详情    │
└─────────────────┬───────────────────┘
                  ↑ 用户点击 / 快捷键唤起
┌─────────────────┴───────────────────┐
│          Layer 1 · 折叠态           │  零注意力
│         悬浮入口 / 角标 / 摘要      │
└─────────────────────────────────────┘
```

**层转换原则：**
- Layer 1 → Layer 2：用户主动触发，或系统在允许条件下触发
- Layer 2 → Layer 3：Escalation（携带上下文）或 Navigation（用户主动跳转）
- Layer 2 → Layer 1：任务完成、用户关闭、超时自动收起
- **不存在** Layer 1 → Layer 3 的直接跳转

### 3.2 Layer 1 · 折叠态内容结构

| 元素 | 显示条件 | 内容上限 |
|------|----------|----------|
| 图标 | 始终显示 | — |
| 角标数字 | 有未读通知时 | 最多显示 99+ |
| 摘要文字 | 有待处理事项时 | 最多 20 字 |
| 状态色 | Agent 执行中 / 错误态 | 仅三种：默认 / 执行中 / 错误 |

### 3.3 Layer 2 · 展开态内容结构

**展开态尺寸约束：**

| 属性 | 数值 |
|------|------|
| 宽度 | 固定 360px |
| 最小高度 | 480px |
| 最大高度 | 屏幕高度 × 80% |
| 内容区超出时 | 内部滚动，顶部栏与输入区固定 |

**展开态内容区：统一会话模式**

Widget 展开后为单一会话界面，不再区分独立的会话/任务/通知模式。所有交互通过消息流完成，Agent 在会话中直接处理问答、轻任务、通知响应。

```
统一会话模式
  用于：快速问答、多轮对话、轻任务创建与执行、通知查看与响应
```

### 4.1.1 Hover 态高频推荐动作（Copilot 模式）

**设计背景：** 用户不倾向于主动构思复杂指令，最常使用的功能应通过高频点击一键直达。当用户鼠标 Hover 悬停在 Layer 1（折叠悬浮球）多于 `300ms` 时，会自动露出一式极精美、高对比拟物深色底、无庞大卡片边框外框遮挡、宽度不固定且由文字自由撑开的胶囊药丸式（Chips）极简快捷行动按钮组。

**场景感知（Context-Aware）动态推荐逻辑：** Widget 会自动探测用户当前的焦点应用/网页形态（如 Jira 敏捷看板、Outlook 邮件客户端、代码编辑器等），并针对性地推荐该环境下最痛点、最高频的操作，实现 “所见即所想” 的推荐体验。

#### 4.1.1.1 核心场景感知映射表

| 焦点环境 (Host Context) | Hover 呈现胶囊按钮样式 (自支撑宽度) | 推荐动作 | 用户点击后自动发送的指令 | 动作分类 |
|-------------------|--------------|----------|-----------------------|---------|
| **通用场景 (General)** | 无背框 · 精致磨砂胶囊药丸 | 总结当前页面<br/>解释选中内容<br/>生成会议纪要 | 总结当前活动窗口的分析内容并输出3行摘要<br/>解释剪贴板中捕获的选定内容<br/>/todo 自动提取并生成本轮对话日程会议纪要 | 会话内执行<br/>会话内执行<br/>会话内执行 |
| **Jira 表单/看板** | 无背框 · 精致磨砂胶囊药丸 | **写一段测试用例**<br/>**分析阻塞 Block 原因**<br/>**一键认领并拆分 Subtask** | 针对当前选中的 Jira 工单生成一份完整的测试边界用例模板<br/>分析该 Jira 问题中阻碍的根本成因，并提供 3 个可选的恢复路径<br/>/todo 认领当前任务，并在当前 Jira 工单下拆分并指派 3 个子任务表单 | 会话内执行<br/>会话内执行<br/>会话内执行 |
| **Email 邮件客户端** | 无背框 · 精致磨砂胶囊药丸 | **起草专业回复**<br/>**提取邮件待办与日程**<br/>**一键总结超长邮件链** | 总结并在 100 字内提炼发件人的核心意图，起草一封严谨专业的复信草稿<br/>/todo 提炼并创建此邮件中涉及我本人的所有待办时限动作卡片<br/>一键汇总这封超长的多往返邮件链条，归纳三方矛盾争执点与决策线索 | 会话内执行<br/>会话内执行<br/>会话内执行 |
| **Code Editor 窗口** | 无背框 · 精致磨砂胶囊药丸 | **解释选定代码段**<br/>**优化当前片段性能**<br/>**一键为其编写单元测试** | 用白话解释剪贴板内选中代码片段的核心算法实现逻辑与边界异常风险值<br/>审核是否有更低的时间/空间复杂度优化空间，重构输出该代码段<br/>为当前剪贴板内写好的函数自动配齐完整的 Mock 单元测试覆盖率用例 | 会话内执行<br/>会话内执行<br/>会话内执行 |

**交互细节：**
- **自适应一侧贴边**：当悬浮球吸附在屏幕左侧边缘时，胶囊粒子按钮组自动靠左（`items-start`）对齐堆叠；吸附在右侧边缘时自动靠右（`items-end`）对齐堆叠，优雅保留桌面视窗呼吸空间。
- **防滑抖动规则**：Hover 计时必须大于 `300ms`，以防鼠标随手划过时误触发。当鼠标离开 `400ms` 内，若无后续动作，胶囊快捷按钮组自动淡出，体验轻快无打断。
- **1-Click 直达体验**：点击其中任何一个药丸卡片，该按钮组淡出，伴随面板优雅张开并直接执行，免去用户进入后再思考的过程，符合**极致最短操作路径**规则。

#### 4.1.1.2 Hover 快捷操作的完整交互流程

```
折叠态悬浮球 (IDLE)
        ↓
  鼠标进入悬浮球区域
        ↓
  计时器启动（300ms 阈值）
        ├── 300ms 内鼠标离开 → 取消，保持 IDLE
        └── 300ms 到达 ───────────────→ HOVER_ACTIVE
                                        ↓
                              场景检测（识别焦点宿主应用）
                                        ↓
                              胶囊按钮组淡入（200ms ease-enter）
                              吸附侧自适应对齐（左吸靠左 / 右吸靠右）
                                        ↓
                    ┌───────────────────┼───────────────────┐
                    ↓                   ↓                   ↓
              鼠标悬停在按钮上        鼠标移出按钮组        鼠标点击某个按钮
              （按钮高亮态）          离开计时器启动            ↓
                    │              （400ms 阈值）         CLICK_EXECUTE
                    │                   │                    ↓
                    │              400ms 内回到按钮组    按钮组淡出（200ms）
                    │              → 取消离开计时          ↓
                    │                   │              面板展开（300ms ease-spring）
                    │              400ms 到达               ↓
                    │              → 按钮组淡出          自动发送预填指令
                    │              → 回到 IDLE               ↓
                    │                                  进入 THINKING 状态
                    │                                      （见 §5.3）
```

**流程关键约束：**

| 阶段 | 约束 | 说明 |
|------|------|------|
| 场景检测 | 须在 100ms 内完成 | 不得阻塞按钮组渲染 |
| 按钮组淡入 | 200ms `ease-enter` | 不打断用户当前视线焦点 |
| 离开容忍 | 400ms 缓冲 | 防止鼠标微动导致按钮组意外消失 |
| 点击 → 展开 | 总耗时 ≤ 500ms | 按钮组淡出（200ms）+ 面板展开（300ms）并行或串行 |
| 指令发送 | 面板展开后立即执行 | 用户无需在输入框中二次确认 |

**Hover 快捷操作与 Quick Action Bar 的关系：**

| 维度 | Hover 胶囊按钮组 | Quick Action Bar（§4.4.1） |
|------|------------------|---------------------------|
| 出现时机 | 折叠态 Hover > 300ms | 展开态始终可见 |
| 按钮数量 | 固定 3 个 | 2 ~ 4 个，动态调整 |
| 点击行为 | 展开面板 + 自动发送指令 | 在会话中预填或触发上下文 |
| 场景感知 | 基于焦点宿主应用 | 基于焦点宿主 + 用户角色 |
| 消失条件 | 鼠标离开 400ms | 面板收起时消失 |

> **两者互补而非重复**：Hover 胶囊按钮用于用户「还没想好要做什么」的探索态，Quick Action Bar 用于用户「已在会话中」的执行态。

**自动避让规则：**
```
检测到展开态与以下元素重叠时，自动向反方向移动：
  · 系统通知区域（macOS 右上角 / Windows 右下角）
  · 全屏应用边界
  · 任务栏 / Dock

不自动避让：普通应用窗口（用户可自行拖拽调整）
```

**多显示器规则：**
```
Widget 跟随焦点窗口所在显示器
  → 切换焦点到另一显示器时，Widget 随之迁移
  → 迁移时保持相对位置（同侧、同高度比例）

全屏模式：Widget 折叠态隐藏，退出全屏后恢复至原位置
```

**虚拟桌面规则：**
```
默认：Widget 跟随用户，在所有虚拟桌面可见
可选：用户可将 Widget 固定在特定虚拟桌面（企业策略可强制）
```

### 4.5 最短操作路径约束

> **操作步骤计算口径：** 唤起动作（1 步，固定）+ 任务执行步骤（≤ 3 步）= 用户总操作 ≤ 4 步。Step 6.3 Agent 判定规则中的「操作步骤 ≤ 3 步」不含唤起，两者口径一致。

| 用例 | 最短路径 | 步骤上限 |
|------|----------|----------|
| UC-01 快速问答 | 唤起 → 输入 → 回车 | 3 步 |
| UC-04 创建待办 | 唤起 → 输入 "创建待办xxx" → 回车 | 3 步 |
| UC-07 会议提醒响应 | 点击角标 → 点击行动按钮 | 2 步 |
| UC-10 Escalation | 唤起 → 输入 → 回车 → 点击「打开主应用」 | 4 步 |
| UC-12 不可逆拦截 | 唤起 → 输入 → 回车 → 点击「打开主应用」 | 4 步 |

**原则：Widget 内任何操作不超过 4 步，超过 4 步的任务应触发 Escalation。**

### 4.6 多 Widget 实例管理

```

  每个账号对应一个独立 Widget 实例
  → 多实例同时存在时，按账号色区分折叠态图标
  → 同时展开：垂直排列，间距 8px
  → 焦点切换：点击对应实例，其余自动折叠（可配置）

层叠顺序：最近活跃的实例始终在最顶层
```

### 4.7 安全与权限交互规则

| 能力 | 授权提示时机 | 撤销入口 |
|------|-------------|----------|
| 剪贴板读取 | 首次使用前弹出系统权限请求 | 系统设置 + Widget 设置页 |
| 截图 | 每次使用前确认（不记住） | — |
| 麦克风 | 首次使用前弹出系统权限请求 | 系统设置 + Widget 设置页 |
| 通知推送 | 首次启动时请求 | Widget 设置页 |

**权限被拒绝时的降级处理：**
```
剪贴板权限被拒：隐藏自动读取功能，不影响手动输入
截图权限被拒：截图入口置灰，Tooltip 提示「请在系统设置中开启」
麦克风权限被拒：语音输入入口隐藏
```

## Step 5 · 状态体验与生命周期

### 5.0 设计原则

```
1. 状态可感知：用户始终知道 Widget 正在做什么
2. 状态可恢复：任何中断都有明确的恢复路径
3. 状态不阻塞：错误与异常不占用用户主注意力
```

### 5.1 状态全景

**Widget 自身状态（层级维度）**

```
折叠态：默认 / 有未读（角标）/ 执行中（动效）/ 错误（色彩提示）

展开态：空态 / 输入中 / 加载中 / 响应中 / 完成态 /
        Escalation 提示态 / 错误态 / 离线态 / 权限异常态
```

**Agent 任务状态（任务维度）**

```
未开始 / 进行中（后台执行）/ 等待用户输入 / 完成 / 失败 / 超时
```

### 5.2 空态

**首次空态（新用户）**

- 展示最多 3 个 P0 用例快捷入口
- 点击快捷入口直接预填输入框，不跳转页面
- 使用过 3 次后快捷入口隐藏

**回归空态（会话结束后）**

- 完成状态保留 3 秒后过渡至回归空态
- 显示上一条任务摘要，提供上下文连续感
- 不主动推荐下一个任务

### 5.3 加载态

**阶段 1 · 等待首字响应（0 ~ 首字出现）**

| 时间节点 | 行为 |
|----------|------|
| 超过 1 秒 | 三点动效出现 |
| 超过 3 秒 | 三点动效下方出现「仍在处理中…」 |
| 超过 10 秒 | 进入超时处理流程 |

**阶段 2 · 流式输出**

- 流式输出期间输入框禁用，显示「回复中…」
- 用户可点击「停止」中断输出
- 中断后保留已输出部分，显示「已停止」标记
- 输出完成后输入框立即恢复可用

### 5.4 完成态

**轻任务完成：** 在消息流中以 InlineCard 展示完成确认 + 任务标题回显 + Navigation 查看入口  
**问答完成：** 无特殊完成态，光标动效消失即为完成信号  
**通知响应完成：** 通知卡片标记为已处理，自动收起或保留在消息流中  
**后台任务完成：** 折叠态角标 → 展开后在消息流中显示完成 InlineCard + 查看结果 / 关闭

### 5.5 错误态

**Level 1 · 局部错误（不影响 Widget 整体可用性）**
- 错误提示内联在消息下方，提供「重试」
- 连续失败 3 次后升级为 Level 2

**Level 2 · 会话级错误（当前会话不可用）**
- 内容区展示错误说明 + 「开始新会话」
- 输入框禁用

**Level 3 · Widget 级错误（整体不可用）**
- 折叠态图标变为错误色
- 展开后提供「重试」+ 「打开主应用」
- 错误信息使用用户语言，错误代码仅在「查看详情」后可见

### 5.6 超时处理

```
0s ──── 1s ──── 3s ──── 10s ──── 30s
        ↓        ↓        ↓         ↓
     三点动效  「仍在处理」 超时提示  强制终止
                         + 重试选项  + Level 2 错误
```

- 用户选择「继续等待」：重置计时器，再等 20 秒
- 用户选择「取消」：终止请求，返回空态
- 30 秒强制终止：自动进入 Level 2 错误态

### 5.7 离线态

| 功能 | 离线时行为 |
|------|-----------|
| 查看历史消息 | 正常可用（本地缓存） |
| 发送新消息 | 禁用，提示离线 |
| 通知接收 | 暂停，恢复后补推 |
| 进行中的 Agent 任务 | 继续在服务端执行，恢复后同步状态 |

**网络恢复后：** 离线标签消失、输入框恢复、自动同步任务状态、若有任务完成则角标提示。

### 5.8 权限异常态

- 不弹出系统级警告，在 Widget 内内联提示
- 提供替代操作路径（如「手动粘贴」）
- 「前往设置」直接跳转系统权限设置页
- 同一权限问题在同一会话内只提示一次

### 5.9 上下文失败态

- Widget 正常可用，上下文相关功能静默降级
- 仅在用户主动使用上下文功能时提示
- 提示示例：「无法读取当前页面内容，你可以手动描述需要处理的内容」

### 5.10 生命周期管理

**进程挂起 / 系统休眠唤醒：**
```
挂起前：保存会话状态（会话 ID、输入草稿、滚动位置），暂停 WebSocket
唤醒后：恢复 WebSocket，检查任务状态变化，恢复输入草稿
       若离线：进入离线态；若在线：重建连接，同步状态
```

**Widget 更新：**
```
不强制重启，折叠态显示更新提示点
用户主动点击后提示「重启以更新」
用户选择时机，不打断当前任务
```

## Step 6 · Agent 行为

### 6.0 设计前提

Widget 内的 Agent 行为设计核心矛盾：

```
用户希望 Agent 直接完成任务
          VS
Widget 的能力边界决定部分任务必须升级
```

**目标：在不打断用户的前提下，尽可能在 Widget 内完成任务；在必须升级时，让升级过程无感且无损。**

### 6.1 Agent 在 Widget 内的角色定义

| 职责 | 说明 |
|------|------|
| 意图识别 | 判断用户输入属于哪类任务 |
| 边界判断 | 评估任务是否可在 Widget 内完成 |
| 轻任务执行 | 直接完成 Widget 范围内的任务 |
| Escalation 决策 | 判断何时、以何种方式建议升级 |
| 上下文打包 | Escalation 时完整传递当前上下文 |

### 6.2 意图识别与任务分类

| 类型 | 判定条件 | Agent 行为 |
|------|----------|-----------|
| A · 信息获取 | 预估输出 ≤ 500 字，无需执行操作 | 直接回答 |
| B · 轻任务执行 | 操作步骤 ≤ 3 步，完成时间 < 2 分钟 | 在会话中直接执行，以内联卡片反馈结果 |
| C · 超出边界 | 触发任意软限制 Escalation Heuristic | 建议升级（软） |
| D · 模糊意图 | 无法确定任务类型或关键参数缺失 | 最小化澄清 |
| E · 不可逆操作 | 涉及批量删除、权限变更、资金、发布 | 强制升级（硬） |

### 6.3 Widget 内可完成任务的判定规则

**可在 Widget 内完成（全部条件须满足）：**

```
✓ 预估完成时间 < 2 分钟
✓ 输出内容 ≤ 500 字
✓ 操作步骤 ≤ 3 步（不含唤起）
✓ 不涉及不可逆操作
✓ 不需要展示超过 3 个并列信息模块
✓ 不依赖完整历史上下文
✓ 不需要精确编辑长文本
```

### 6.4 模糊意图处理

**澄清原则：能推断的不问，能给默认值的不问，必须确认的只问一个问题。**

**Level 0 · 直接执行（有合理默认值）**
```
用户：帮我创建一个待办
Agent：直接创建，标题取关键词，完成后显示「待办已创建：[标题]」，提供「修改」入口
```

**Level 1 · 内联澄清（缺少关键参数）**
```
用户：提醒我
Agent：提醒你什么？
```

**Level 2 · 选项澄清（意图模糊但有限选项）**
```
用户：发一下
Agent：你想发送什么？
      [消息]  [文件]  [报告]
```

**禁止行为：** 一次提出超过 1 个澄清问题；对有默认值的参数澄清；澄清后再次确认。

### 6.5 Escalation 决策规则

#### 6.5.1 触发时机

Escalation 建议须在**响应开始前**给出，禁止先执行一半再提示升级。

#### 6.5.2 软限制话术规则

须包含三要素：说明原因 + 给出继续选项 + 保证上下文不丢失。

**标准模板：**
```
[原因]，建议在主应用中完成，[收益说明]。

[继续在这里开始]    [打开主应用]
```

**禁止话术：**
```
✕ 「这个我做不到」
✕ 「请前往主应用」
✕ 「此功能不在 Widget 中」
```

#### 6.5.3 硬限制话术规则

须包含两要素：说明这是不可逆操作 + 只提供「打开主应用」。

**标准模板：**
```
[操作类型]属于不可逆操作，请在主应用中确认后执行。

[打开主应用]
```

#### 6.5.4 用户拒绝 Escalation 后的行为

- Agent 正常执行，本轮会话内不重复 Escalation 建议
- 若输出超出显示能力：展示前 500 字 + 「查看完整内容」Navigation 入口

### 6.6 多轮对话中的 Agent 行为

**轮次追踪：**
```
1 ~ 3 轮：正常执行
第 4 轮：评估是否需要 Escalation
  → 仍在边界内：继续，不提示
  → 已超出边界：软限制 Escalation
第 7 轮：无论任务类型，给出一次 Escalation 建议，用户拒绝后不再重复
```

**上下文窗口管理：**
- Widget 上下文窗口为最近 10 条消息
- 超出后提示用户（仅一次）：「早期对话内容已不在当前上下文中，如需参考请打开主应用查看完整记录。」

### 6.7 后台任务的 Agent 行为

```
发起 → 显示「任务已开始」+ 任务 ID → Widget 可收起
→ 折叠态执行中动效 → 完成后角标提示
→ 展开查看摘要 → 「查看完整结果」Navigation 至主应用
```

**失败处理：** 自动重试最多 2 次（用户无感知），2 次后提示用户手动重试。

### 6.8 Agent 响应格式规则

| 格式元素 | Widget 内规则 |
|----------|--------------|
| 段落 | 最多 3 段，每段 ≤ 3 行 |
| 列表 | 最多 5 条，超出折叠 |
| 代码块 | 支持，超过 20 行触发 Escalation 建议 |
| 表格 | 不渲染，转为文字描述或 Escalation |
| 图片 / 图表 | 不渲染，提供「在主应用中查看」入口 |

**响应长度控制：**
```
目标长度：≤ 150 字
超过 300 字：自动生成摘要（≤ 100 字）+ 「查看完整回答」Navigation
超过 500 字：触发软限制 Escalation 建议
```

**禁止响应模式：**
```
✕ 开场白（「好的，我来帮你…」）
✕ 结束语（「希望这对你有帮助！」）
✕ 重复用户的问题
✕ 不必要的分段和小标题
```

## Step 7 · 通知策略

### 7.0 设计前提

```
通知的价值 = 信息相关性 × 送达时机
通知的代价 = 打断用户专注状态的成本
```

### 7.1 通知分级体系

| 级别 | 名称 | 表现形式 | 适用场景 |
|------|------|----------|----------|
| L1 | 静默通知 | 仅更新折叠态角标数字 | 低优先级信息更新 |
| L2 | 轻提示 | 折叠态摘要文字 + 角标，无声音 | 中优先级，需知晓但不急迫 |
| L3 | 强提醒 | Widget 展开，内容直接呈现，可选声音 | 高优先级，需立即响应 |

**分级判定矩阵：**

| 事件类型 | 默认级别 | 可降级 | 可升级 |
|----------|----------|--------|--------|
| 后台任务完成 | L2 | ✓ → L1 | ✓ → L3（失败时） |
| 会议提醒（> 15 分钟）| L1 | — | — |
| 会议提醒（5 ~ 15 分钟）| L2 | ✓ → L1 | — |
| 会议提醒（< 5 分钟）| L3 | ✓ → L2 | — |
| 紧急 @ 提及 | L3 | ✓ → L2 | — |
| 普通 @ 提及 | L2 | ✓ → L1 | — |
| Agent 任务失败（需介入）| L3 | — | — |
| Agent 任务失败（可自动重试）| L1 | — | — |
| 权限即将过期 | L2 | ✓ → L1 | ✓ → L3（已过期）|
| 系统状态更新 | L1 | — | — |

### 7.2 各级通知的视觉与交互规范

**L1 · 静默通知：** 角标数字更新，无动效、无声音、不展开。用户点击角标后，Widget 展开并在消息流顶部以 InlineCard 展示最近通知列表。

**L2 · 轻提示：** 摘要文字从右侧滑入（200ms），停留 4 秒后收起，角标保留。用户点击角标后，Widget 展开并在消息流中展示通知 InlineCard。

**L3 · 强提醒触发条件（须全部满足）：**
```
✓ 用户当前状态 ≠ 专注模式 / 全屏 / 演示
✓ 该类通知未被用户或企业策略静默
✓ 距上次 L3 通知 > 30 秒（D-02 确认值）
```

**L3 展开行为：** Widget 展开，通知以 InlineCard 形式插入消息流顶部，用户可直接在会话中操作行动按钮。

**通知 InlineCard 结构：**
- 来源标签 + 通知标题（≤ 20 字）+ 通知摘要（≤ 40 字）
- 主行动按钮 + 次行动按钮（最多 2 个）
- 「忽略」文字链（始终存在，不可隐藏）

**声音规则：** 默认无声音，用户可在设置中开启，企业策略可强制关闭。

### 7.3 通知合并规则

```
5 分钟内同一来源的同类通知合并为一条，更新摘要
跨类型通知不合并
合并通知展开后显示列表，支持逐条处理或「全部标为已读」
```

### 7.4 免打扰机制

**即时静默（快捷操作）：**
- 「静默 1 小时」/ 「静默今天剩余时间」/ 「进入专注模式」

**专注模式（D-01 已确认）：**
```
开启来源：
  · 用户在 Widget 内手动开启
  · 系统 API 联动（macOS Focus Mode / Windows Focus Assist）
  · 日历检测到正在开会（可选，需日历权限）
判定：任意一个条件为真即进入专注模式

开启后：L3 → L2，L2 → L1，L1 不变
结束后：积压通知不补推 L3，仅更新角标
```

**企业策略控制：**

| 策略项 | 用户可否覆盖 |
|--------|-------------|
| 强制关闭声音 | 否 |
| 禁止特定通知类型 | 否 |
| 强制 L3 降级 | 否 |
| 设定通知时间窗口 | 可收窄，不可扩大 |

### 7.5 通知的时机策略

```
深度专注中   → 延迟推送至下一个任务间隙
轻度专注中   → 按默认级别推送

延迟推送上限：
  L1：无上限
  L2：最多延迟 10 分钟
  L3：最多延迟 2 分钟；超时仍无合适时机：强制推送

距事件发生 < 2 分钟的 L3：忽略用户状态，立即推送
  （例外：全屏 / 演示模式仍静默）
```

### 7.6 通知的生命周期

```
通知产生 → 分级 → 时机判断 → 推送
  ├── 点击行动按钮 → 执行 → 标为已处理
  ├── 点击「忽略」 → 标为已忽略，不再提醒
  ├── 未响应（L3）→ 60 秒后自动收起，降级为 L2 角标
  └── 专注模式中  → 积压，结束后以 L1 呈现

已处理通知保留在通知历史，Widget 内可查看最近 10 条
```

### 7.7 通知与 Escalation 的交叉规则

通知行动按钮触发 Escalation 的条件：属于不可逆操作 / 需要完整上下文 / 步骤超过 4 步。

```
✓「在主应用中审批」  ← 直接 Escalation
✕「审批」→ 执行一半 → 「请在主应用中完成」
```

### 7.8 通知设计的禁止行为

```
✕ 同一事件 5 分钟内推送超过 1 次
✕ 全屏 / 演示模式下触发任何可见通知
✕ 使用模糊摘要（「你有新消息」）
✕ 通知内容超过 2 行才能理解
✕ 行动按钮超过 2 个
✕ 隐藏「忽略」选项
✕ 通知声音默认开启
```

## Step 9 · 数据闭环

### 9.0 五个核心问题

```
1. 用户愿意用 Widget 吗？          → 唤起指标
2. Widget 帮用户完成任务了吗？      → 完成指标
3. Widget 把用户推走了多少次？      → Escalation 指标
4. 用户在哪里放弃了？              → 转化漏斗
5. 哪些体验让用户感到挫败？        → 负反馈指标
```

### 9.1 埋点方案

**埋点原则：**
- 最小必要：只埋回答核心问题必须的事件
- 隐私：不记录消息内容，只记录结构特征（长度、类型、轮次）
- 一致性：严格遵循 Step 8.5 命名规范

**基础事件属性：**

```typescript
interface BaseEventProperties {
  event_name: string;
  timestamp: number;
  session_id: string;
  instance_id: string;
  platform: 'macos' | 'windows';
  widget_version: string;
  display_state: 'collapsed' | 'expanded';
}
```

**埋点优先级：**

P0（必须）：`widget.entry.expanded` / `widget.entry.collapsed` / `widget.session.started` / `widget.session.message_sent` / `widget.session.response_received` / `widget.escalation.prompted` / `widget.escalation.accepted` / `widget.escalation.dismissed` / `widget.task.completed` / `widget.task.failed` / `widget.feedback.negative`

P1（重要）：`widget.hotkey.triggered` / `widget.session.interrupted` / `widget.notification.*` / `widget.error.*` / `widget.offline.*` / `widget.feedback.escalation_regret`

P2（扩展）：`widget.notification.expired` / `widget.task.background_started` / `widget.error.recovered`

### 9.2 核心指标体系

**北极星指标：任务完成率**

```
定义：Widget 内闭环完成的任务数 ÷ Widget 内发起的总任务数
健康基线：> 65%
预警阈值：< 50%
```

**唤起指标：**

| 指标 | 定义 | 健康基线 | 预警阈值 |
|------|------|----------|----------|
| 日活唤起率 | 当天至少唤起一次的用户数 ÷ 安装用户数 | > 40% | < 20% |
| 快捷键习惯养成率 | hotkey 触发占 expanded 总数比例 | 7 日内 > 30% | — |

**转化漏斗（以 UC-01 为例）：**

```
唤起 → 开始输入 → 发送 → 收到响应 → 自然收起
100% → 85%     → 80% → 75%      → 60%

断层分析：
  Step 2 流失大 → 空态设计有问题
  Step 3 流失大 → 输入体验有问题
  Step 4 流失大 → 响应质量或速度有问题
```

**Escalation 指标：**

| 指标 | 健康范围 | 过低含义 | 过高含义 |
|------|----------|----------|----------|
| 触发率 | 5% ~ 20% | 任务过于简单或边界过宽 | 边界过严或预期不匹配 |
| 接受率 | 40% ~ 70% | Escalation 过于激进 | 边界设置合理或过严 |
| 后悔率 | < 10% | — | Escalation 不准确 |

**健康指标：**

| 指标 | 健康基线 | 预警阈值 |
|------|----------|----------|
| L3 错误率 | < 1% | — |
| 超时率（latency > 10s）| < 3% | > 8% |
| 响应负反馈率 | < 5% | > 10% |
| 离线发生率 | < 5% | > 15% |

### 9.3 指标异常响应优先级

| 指标 | 预警阈值 | 响应优先级 | 首要排查方向 |
|------|----------|-----------|-------------|
| 任务完成率 | < 50% | P0 | 转化漏斗断层 |
| L3 错误率 | > 1% | P0 | 服务端稳定性 |
| 超时率 | > 8% | P0 | Agent 响应性能 |
| DAU 唤起率 | < 20% | P1 | 空态与唤起入口 |
| Escalation 后悔率 | > 20% | P1 | 判定准确度 |
| 响应负反馈率 | > 10% | P1 | Agent 响应质量 |
| 离线发生率 | > 15% | P2 | 网络与服务稳定性 |

